# Example：三工具选型 × Pre-flight 环境侦察（技术约束驱动的分析重构）

> 版本：2026-04-11
> 适用场景：技术工具/平台选型类议题，候选方案中存在「根本不可用」的选项
> 价值：展示 pre-flight 环境侦察颠覆分析方向的典型案例，以及多轮迭代 brainstorm 的继承模式

---

## 背景

议题是「在 Claude Code 会话中，GitHub 推送操作应使用 MCP GitHub插件、git CLI 还是 gh CLI？」，涉及三个候选工具的分工评估。session 分为两轮：

- 第一轮：用户未提供工具安装状态，编排层主动执行环境侦察，发现 gh CLI 未安装
- 第二轮：用户决定引入 gh CLI，要求评估三工具分工

---

## 亮眼表现一：Pre-flight 侦察改变分析前提

**触发时机**：议题涉及工具选型，用户提及「gh CLI」但未说明是否已安装

**编排层的动作**：
```bash
which git && git --version   # → git v2.43.7 已安装
which gh                     # → not found
```

**如果没有侦察会发生什么**：四个角色会在「MCP vs git vs gh 三选一」的框架下分析，其中 gh 相关的建议和分析全部基于一个不存在的工具，结论毫无执行价值。

**实际结果**：侦察结果「gh CLI 未安装」注入所有 agent prompt 后，角色分析自动聚焦在「MCP vs git」的实际二选一，每条建议都可以立即执行。

**对 skill 的启示**：技术工具类议题中，候选方案的「物理可用性」是先于价值判断的前置约束。环境侦察应在 agent 启动前完成，而非依赖用户申报。

---

## 亮眼表现二：域内事实注入让角色分析脱离教科书层次

**触发时机**：自定义角色「AI助手效率专家」需要分析两种工具的 token 消耗差异

**注入到 prompt 的具体事实（而非框架）：**
- `gh pr list --json number,title,state --jq '.[0:5]'` 可精确裁剪输出
- MCP 返回完整 JSON 对象含 `node_id`、`_links` 等冗余字段，不可裁剪
- MCP schema（40+ 工具）已常驻 context，是沉没成本而非变动成本

**角色产出的有力要点（因为有事实支撑）：**
> "Schema 已是沉没成本，边际调用成本才是决策变量。真正的增量成本是每次 MCP 调用的 tool result tokens——与 Bash 调用 git CLI 的 stdout tokens 量级相当，MCP 在此维度无额外劣势。"

> "gh CLI 配合 `--json --jq` 裁剪后 token 量级是几十，MCP 完整返回动辄几百 token——在高频 GitHub 操作的会话中，累积 context 增长速度差距显著。"

**如果没有事实注入会产出什么**：「MCP 会消耗更多 token，gh CLI 更轻量」——这是任何人都能说的原则性判断，没有量级，无法驱动决策。

**对 skill 的启示**：技术角色的 prompt 中，「关键技术细节（分析时必须引用）」小节的作用是给角色「弹药」——有了具体命令和数据，角色才能产出「只有查过文档的人才会说」的内容。

---

## 亮眼表现三：迭代追问继承上轮结论，避免重走旧地

**触发时机**：第一轮 brainstorm 完成后，用户说「引入 gh CLI 后再讨论一轮」

**第二轮 agent prompt 的继承写法：**
```
【上一轮已确定的结论，本轮无需重复论证】
- MCP 适合 GitHub API 操作（PR、issue），git 适合本地操作
- MCP schema 已是沉没成本，不构成避免使用的理由
- gh CLI 已被认定为值得引入（用户主动决策）

【本轮新增变量】
gh CLI 引入后，三工具如何合理分工？是否存在冗余？
```

**实际效果**：第二轮角色分析没有重复「MCP 好还是 git 好」的讨论，直接聚焦在三工具的边界划分和凭证统一方案，输出密度显著提升。

**对 skill 的启示**：多轮 brainstorm 的关键不是「每次都从零开始」，而是「继承已定结论，只分析新增量」。继承写法应明确标注「无需重复论证」，避免角色因谨慎而重复已完成的分析。

---

## 三个模式的触发信号总结

| 模式 | 触发信号 | 动作 |
|------|---------|------|
| Pre-flight 侦察 | 议题含「工具选型」「平台迁移」「配置评估」 | 用 Bash 验证工具存在性和当前状态，注入所有 agent prompt |
| 事实注入 | 自定义角色为技术领域专家 | prompt 中增加「关键技术细节（分析时必须引用）」小节，含具体命令和数据 |
| 迭代追问 | 用户说「基于上轮」「加入 X 后」「再讨论」 | 新一轮 prompt 明确标注已定结论和本轮新增变量 |
