# Example：Harness 架构 × 组织落地分析（domain practitioner 角色涌现）

> 版本：2026-04-08  
> 适用场景：技术方案涉及组织推行、跨部门协作改造、管理层 buy-in 等**组织变革类议题**  
> 价值：展示「领域从业者」自定义角色填补 default 4 roles 盲区的场景，提炼第5角色模式的通用触发规则

---

## 背景

议题是「研发全管线 Harness 架构设计」，用户提出了5个涉及组织推行的核心论点（Memory Governance、泳道模型、强制 handoff 等）。除 default 3 个角色（AI Infra Builder / Challenger / PM）外，用户额外引入了「游戏制作人视角」——一位管理过 30-100 人游戏研发团队的制作人。

这个角色并不在 brainstorm 预设角色库中，是本次 session 的**涌现亮点**。

---

## 亮眼表现一：Schema 标准化成本的组织现实接地

**触发时机**：技术方案预设了"跨部门 schema 统一"作为已知前提

**default 角色的盲区**：AI Infra Builder 会指出 schema 版本漂移的工程风险，但不会质疑 schema 能不能统一；Challenger 会质疑方案假设，但缺乏领域数据支撑

**游戏制作人的补充**：
> "跨部门 schema 标准化是 3-6 个月的组织变革，每个团队的格式各异。把这个前置条件当已知条件来设计 harness，会在第一步就卡住。"

**为什么有力**：给出了具体时间量级（3-6 个月），把"技术假设"转化为"组织工程量估算"，让抽象风险变得可衡量、可讨论

**类推规则**：当方案预设了跨职能的格式/流程统一时，domain practitioner 角色应给出该统一工作在真实团队中的工程量量级，而不只是说"这很难"。

---

## 亮眼表现二：强制机制的组织行为预判

**触发时机**：方案包含"强制弹窗""强制 handoff""强制提交"等机制设计

**default 角色的盲区**：Challenger 会质疑"人愿不愿意填"，但更多从假设层面；PM 会建议降低摩擦，但缺乏真实团队行为数据

**游戏制作人的补充**：
> "游戏研发的节奏是 feature 做完立刻切下一个，没有人愿意在中间多一个强制步骤。除非 handoff 文档能直接替代现有的会议纪要或 Confluence 页面，否则团队会绕过它。必须做成现有流程的自然替代，而不是附加工序。"

**为什么有力**：不是在猜测"可能有阻力"，而是描述了这个行为的具体形态（绕过 = 还是切下一个任务），并给出了对策的必要条件（替代而非附加）

**类推规则**：当方案包含"强制"类机制时，domain practitioner 应描述目标群体绕过该机制的具体行为路径，而不只是说"会有阻力"。

---

## 亮眼表现三：向管理层 ROI 说明的包装重构

**触发时机**：方案的终极落地需要管理层立项/预算审批

**default 角色的盲区**：PM 会关注用户痛点和 MVP；但对"如何向 CEO 立项"的语言体系转换，PM 关切的是用户，不是管理层

**游戏制作人的补充**：
> "向 CEO 或制片人解释 Memory Governance 的 ROI 极难量化。更务实的切入点是包装成'项目知识不随人员流失'——这个痛点管理层有感知，而不是从架构层讲 memory governance，否则会在立项阶段死掉。"

**为什么有力**：提供了具体的语言替换——把"memory governance"替换成"知识不随人员流失"，是一个可以直接写进 PPT 的重构，而不是原则性建议

**类推规则**：当方案需要向非技术管理层立项时，domain practitioner 应提供"管理层语言体系"的具体翻译，而不只是说"需要 ROI"。

---

## 亮眼表现四：自助机制的可行性边界

**触发时机**：方案设计了"由用户/职能部门自行提交 criteria/标准/规则"的自助机制

**default 角色的盲区**：PM 会评估摩擦点；Challenger 会质疑价值；但都不会给出"自提质量会是什么样"的领域经验

**游戏制作人的补充**：
> "让策划或美术自己写验收标准，现实中要么交的是废话（'画风统一、逻辑合理'），要么根本不交。可行路径是制作人先建模板，各职能填空确认，而不是从零提交。"

**为什么有力**：给出了两种具体失败模式（废话 / 不交），直接证伪了"自提"方案，同时给出了可行替代（模板 + 填空）

**类推规则**：当方案依赖"非技术职能自行提交结构化内容"时，domain practitioner 应枚举该群体的具体失败模式，并给出最低可行替代方案。

---

## 为什么这次 session 有效：第5角色的涌现价值

| 维度 | Default 4 roles 覆盖 | 游戏制作人补充的维度 |
|------|---------------------|-------------------|
| 技术可行性 | ✅ AI Infra Builder | — |
| 假设检验 | ✅ Challenger | — |
| 用户交付 | ✅ PM | — |
| 代码实现 | ✅ Coder | — |
| **组织推行阻力** | ❌ 缺失 | ✅ 具体行为路径 |
| **管理层 ROI 语言** | ❌ 缺失 | ✅ 语言体系转换 |
| **工程量现实量级** | ❌ 定性 | ✅ 时间量级估算 |
| **自助机制可行性边界** | ❌ 缺失 | ✅ 失败模式枚举 |

---

## 对 brainstorm skill 编排层的启示

- **第5角色的触发信号**：方案中出现"推行""采纳""强制""全团队""管理层""立项"等词时，建议用户考虑加入该议题领域的资深从业者角色
- **domain practitioner 角色的定义原则**：不是功能性角色（PM/工程师），而是在特定领域管理过真实团队的人——他们的价值在于"这在我们行业里是怎么死的"的具体案例
- **不要泛化为固定角色**：游戏制作人不适合成为 default role，因为它是领域绑定的；价值在于学会"当我需要组织现实接地时，我应该召唤这个议题领域的从业者视角"这个模式
