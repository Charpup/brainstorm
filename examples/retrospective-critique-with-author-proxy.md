# Example: 读后感对抗性批判（原作者代言 + 工作区落地 + 盲区挑战）

> **Pattern 8**：读后感对抗性批判模式的完整复盘。
> **来源**：2026-04-18 session — Bob 读腾讯 KM《AI 能记住全世界，凭什么不能记住你自己？》，已写 5 点初版读后感（锚点），想做对抗性批判找盲区，而非求共识。
> **价值**：展示"原作者代言"作为专业角色类型 + 对抗性批判模式 + Meta-judgment gap 作为 bad 反例。

---

## 议题背景

用户（Bob）读了一篇二次创作文章 Y（腾讯 KM 文章 "KV 个人知识库编译层"）。文章明确声明 inspired by 已知方法论 X（Karpathy 的 LLM wiki）。

用户写了 5 点初版读后感作为锚点（A1-A5），然后明确要求：

> "用 /brainstorm 进行批判，看看我的 insight 有没有错误或者短视的地方，**但不要修正我的 insight，而是你提出有冲突的观点，制造观点之间的张力**。"

这是典型的**对抗性批判模式**——不求建议 / 不求综合 / 不求共识，而是**刻意暴露盲区和归因偏差**。

---

## 角色选择决策（3 角色聚焦而非默认 4）

### 为什么不用默认 4 角色

| 默认角色 | 对本议题的适配度 | 说明 |
|---------|-----------------|------|
| AI Infra Builder | **保留但重塑** | 改为"KV 在 Bob 工作区的落地冲突"视角（非通用 harness 架构评估） |
| Challenger | **保留** | 天然适合攻击 Bob 读后感的思维惯性和归因偏差 |
| Product Manager | **移除** | Bob 自己就是 PM 视角，加 PM 会自我重复 |
| Coder | **移除** | 议题是架构思想而非代码实现，Coder 会被迫抽象发散 |

### 3 个 orthogonal 攻击面（非同一事物不同侧面）

| 角色 | 攻击面 | 关切的成本 |
|------|-------|-----------|
| **Challenger** | Bob **读后感层**（comparing mode、归因密度、完整跳过的章节） | 读者认知成本（下次读类似文章会再犯） |
| **AI Infra Builder** | Bob **工作区落地层**（KV 映射到现有 KB/MEMORY/learning 的冲突） | 能力缺口机会成本（不吸收 → 下次蒸馏必然漂移） |
| **原作者 X 代言人**（Karpathy 视角）| 文章与原方法论的 **范式对比层**（文章是否偏离 X 原旨） | 路线选择隐形成本（走 Y 的路线意味着接受什么前提） |

这三个层是正交维度（读者视角 / 用户设备视角 / 方法论源头视角），**分歧本质是"关心不同层面的成本"**，天然避免同质化。

---

## 特色：原作者代言（Original Author Proxy）作为专业角色

**什么是原作者代言**：不是"生造一个专家"（如"架构师"、"效率专家"），而是让**被文章启发的原方法论发明人亲自下场**评价文章是否偏离原旨。

### 立场注入模板（以 Karpathy LLM wiki 为例）

```
你的角色：Andrej Karpathy（LLM wiki 原始方法论提出者视角）

你的立场与核心哲学：

1. "Few large files beat many small files" — 反对过度分类和深度目录树
2. Flat structure + simple markdown + human readable first
3. Grep-first retrieval — 反正 grep 能找到，不需要预先建索引
4. 反对过度 organizing — 时间花在维护 schema 上损失 > 收益
5. 朴素 + 第一性原理 — LLM wiki 本质是"结构化单向输入 prompt 给 LLM"的素材
6. 不搞 agent 自动化维护 — LLM wiki 是人手工 append，信任人工判断胜过 agent 判断
```

**核心注入原则**：
- 把方法论 X 的**5-8 条核心哲学**作为立场前言
- 让 agent 用**第一人称**（"我"= 原作者）说话
- 要求 agent **原教旨主义**表态——不骑墙，不做"部分同意 + 部分反对"的温吞评价
- 明确要求 agent 和用户锚点直接对话（"Bob 说 X，我的回应是 Y"）

### 为什么有效（产出示例）

原作者代言视角产出了其他角色不会说的句子：
> "这不是'大同小异'，这是**两种根本对立的范式**。你看到的'三层结构'只是表象的相似，底层哲学是反过来的——LLM wiki 里 raw 和 synthesis 混在一起靠 grep 区分；KV 里靠目录结构强制分层。"

> "Why 3 是 KV 架构自己制造的问题，然后设计机制去修。**这是我一直在躲的反模式：用更多自动化去修补前一层自动化带来的问题**。"

这两句只有"LLM wiki 原作者立场"会说——任何通用"架构师"/"挑战者"都不会从这个角度切入。

---

## 对抗性批判的特殊 prompt 指令

每个 agent prompt 末尾都包含一条强制指令：

```
关键约束：
- **不要修正 Bob 的锚点**——你要提出**与锚点冲突的观点**，制造观点张力
- 不要讨论"[议题内容]本身好不好"这种通用问题
- 聚焦用户的读后感：他漏看了什么？他的归因是否精准？他为什么会漏看？
- 明确要求 agent 从指定攻击面切入（见"你的攻击面"段）
```

这与默认 brainstorm 的"给建议 + 求共识"模式根本不同。

## R1 产出质量（每角色要点精选）

### Challenger — 攻读者层

- **Bob 完全没看到原文 Why 2 的 Profile 锚定效应——而这恰恰戳到他自己 persona.md 的心脏**。他 5 条锚点一次都没回到自己身上做 Why 2 自测，因为 comparing mode（LLM wiki vs KV）让他压根没意识到这是在说他。
- **A1 的"大同小异"是判决式先验，不是结论——它压扁了后面 4 条锚点**。把 Y 当 X 的扩展去读，是坐标系错误。
- **A4 的归因比原文低密度一个数量级**——热力学描述（"系统会趋向稳态"）vs 具体故障路径（"二手摘要 → 框架过滤 → 校准失效"）。低密度归因推不出具体修复。
- **Bob 5 条锚点全是 "vs X" 的 diff 模式**——系统性漏掉坐标系外概念。这不是注意力问题，是坐标系外的东西不被 encode。

### AI Infra Builder — 攻工作区落地层

- **Compile 主动防御是 harness-patterns.md 明确缺失的能力**。Bob 的 `/summarize` 当前单向写入，从不回读 archive/ 一手材料——`monetization-models.md` / `bytedance-closed-door-20260410.md` 已存在叙事锁定风险。
- **MEMORY.md 已 206 行 > 200 限**（路由层硬阈值），这就是 KV Lint 会报的 "structural 告警"——但 Bob 没有任何自动检查。务实落地是 `/summarize` pre-check hook + git pre-commit hook（事件驱动 Lint）。
- **Bob 的 `persona.md` 就是 Why 2 批判的"constraint 污染"反面教材**。"核心认知优势" + "迁移价值"是 judgment；叠加 `CLAUDE.md` `@persona.md` 自动载入 = 每次 session 开启就注入锚定。
- **A3 "巡航 agent 集群"在单机部署下跑不起来**，务实方案是事件驱动 hook，不是 always-on 集群。

### Karpathy 视角 — 攻原范式层

- **KV 和 LLM wiki 是两种根本对立的范式**（human-authored vs agent-compiled / flat vs layered / grep-first vs query-first / 低维护 vs 高维护 / 无 profile vs 有 profile）。不是改良关系。
- **KV 的三层分离违反了我的设计理念**。"Few large files beat many small files"——而 KV 做的恰恰是 many directories + many small files + 强制每个 topic 存在三个副本。
- **Why 3 的防御机制是 KV 架构自己制造的问题然后自己设计机制去修**。根源不在 LLM 侧，在架构让 LLM 做了本该由人做的"编译"工作。如果手工 append 根本没这个问题。
- **Why 2 是真洞察，但对 LLM wiki 不构成相关增益**——LLM wiki 里根本没有 Profile 概念。

## Bad 反例：Meta-Judgment Gap

### 现象

3 角色 brainstorm 综合报告完成后，Bob 在"下一轮讨论"时**自己补了 2 个顶层判断**：

1. **范式长期兼容性**：LLM wiki 抽象层少所以对未来模型能力迭代保有兼容性；KV 抽象层多所以强耦合当前模型。短中期出彩但长期可能画蛇添足。
2. **虚假繁荣识别**："32 条决策"、"九维度 Lint" 这种量级本身是来源暴露信号——一周业余时间产 32 条大概率是 agent 合集；对原作者有价值，对外部读者是 context 负担。

**关键事实**：这 2 条**3 个角色都没覆盖**。

### 分析

为什么 3 角色都没覆盖？
- Challenger 关心"Bob 的思维惯性"（读者层），但没跳到**元级**（整个文章这种高维度信号是否值得信任）
- AI Infra Builder 关心"能落地什么"（工作区层），不关心长期兼容性
- Karpathy 视角关心"范式对立"（当下对比），但没跳到**时间维度**（未来模型迭代后 KV 会怎样）

即使 3 角色分化做得好，**抽象归因（范式生命周期、量级信号）仍是 agent 产出的边缘盲点**——agent 擅长在议题给定框架内分析，但不擅长主动"跳出框架做 meta 判断"。

### 对 skill 的启示

brainstorm 综合报告**不是最终答案**。用户对 agent 产出做一次 meta-judgment 补丁是**正常闭环步骤**——不是"brainstorm 失败"，是 agent 能力边界的自然延伸。

编排者应在综合报告末尾**主动提示**用户做这一步。见 SKILL.md §第四步 Meta-Judgment Follow-up。

## 本模式的推广价值

### 1. 识别触发信号

这个模式适用于以下所有情况：
- 用户读了一篇**二次创作文章**（声明 inspired by / 借鉴 / 扩展 了某已知方法论 X）
- 用户已有**初版读后感/锚点**，想做批判性蒸馏而非求共识
- 原方法论 X 的**发明人立场明确**（有公开哲学 / 强立场）

典型场景：
- 读 Anthropic engineering blog 某篇讲 harness 的文章（X = Karpathy LLM wiki 或 Martin Fowler 重构）
- 读别人写的"我如何用 XYZ 框架"（X = XYZ 原作者）
- 读借鉴某本书方法论的二次创作（X = 原书作者）

### 2. 角色搭配的定式

| 角色 | 固定职责 | 注入什么 |
|------|---------|---------|
| 原作者 X 代言人 | 评价文章是否偏离 X 原旨 | X 的 5-8 条核心哲学 + 第一人称 + 原教旨立场 |
| 用户工作区落地师 | 把文章内容映射到用户现有协议 | 用户工作区的关键 file paths + 现有 skill 协议 |
| 读者盲区挑战者 | 攻击用户读后感本身的思维惯性 | 用户锚点的原话 + 要求指出未覆盖章节 + comparing mode 识别 |

### 3. 3 个 orthogonal 层原则

每个角色攻击**不同层级**（而不是同一事物的不同侧面）。层级不重叠 = 分化天然。

### 4. 对抗性批判 prompt 的通用模板

```
你的角色：[角色名]
你的立场：[立场描述，含核心哲学/攻击面]

你需要读的文件：[原文路径 + 用户读后感路径]

用户的锚点（原话）：
[逐条列出，不裁剪]

关键约束：
- 不要修正用户的锚点——提出与锚点冲突的观点，制造观点张力
- 聚焦用户读后感：他漏看了什么？他的归因是否精准？他为什么会漏看？
- 不要讨论原文本身好不好

你的攻击面：[分配给此角色的独占攻击面，3-5 点]

输出要求：
- 3-5 个要点，每条只有你这个角色才会说
- 引用用户锚点原话或原文具体段落，不空谈"他没看到"
- 具象化场景——用户这个盲区在实际工作里会怎样具体出问题
```

## 编排者自检清单

1. **议题是否是对抗性批判**？（用户想找盲区而非求共识）→ 启用 adversarial mode
2. **文章是否是二次创作**？（inspired by 已知 X）→ 引入原作者代言
3. **3 角色攻击面是否正交**？（读者层 / 落地层 / 范式层）→ 避免同质化
4. **每个 agent prompt 是否都含"不要修正，制造冲突"指令**？→ 启用对抗模式
5. **综合报告后是否提醒用户做 meta-judgment**？→ 接受 agent 能力边界

## 与其他模式的关系

- **vs Pattern 3（3 自定义角色不同成本视角）**：Pattern 8 也是 3 角色，但核心区别是**原作者代言**这个独特角色类型；Pattern 3 是"不同成本"，Pattern 8 是"不同层级视角"
- **vs Pattern 7（5 角色外部框架借鉴）**：都涉及"外部方法论评估"，但 Pattern 7 是"是否借鉴整个框架"的决策（5 个对抗视角），Pattern 8 是"读后感批判 + 盲区检测"（3 个分层视角）；Pattern 7 求收敛，Pattern 8 求张力
- **vs Pattern 6（Bad/Good 对比）**：本文件末尾的 Meta-judgment gap 段就是 Bad 示例——可以单独作为质量校准参考
