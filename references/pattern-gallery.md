# 角色搭配模式速查（Pattern Gallery）

> 编排者在第一步（角色选择）时读取此文件，快速判断当前议题适合哪种角色搭配。
> 每个模式从真实 session 提炼，完整案例见 `examples/` 对应文件。

---

## 模式一：领域从业者填补组织盲区
**来源**：`examples/harness-architecture-org-mapping.md`
**触发信号**：议题含"推行""采纳""全团队""管理层立项"
**角色搭配**：默认角色 + 1 个该领域的资深从业者（如游戏制作人）
**核心价值**：从业者给出"这在我们行业里是怎么死的"——具体时间量级、绕过行为路径、管理层语言翻译
**自检**：默认角色能否回答"这个方案在真实团队里推行的工程量是多少"？不能 → 需要从业者。

## 模式二：默认角色迁移到非技术议题
**来源**：`examples/onboarding-strategy-for-ai-pm.md`
**触发信号**：职业策略、对话设计、说服/影响类议题
**角色搭配**：默认 4 角色（但各自做框架迁移）
**核心价值**：PM 把"说服对象"当用户分析；Builder 把"了解现状"压缩为单点探针；Challenger 从用户原文挖反驳证据
**自检**：议题虽非技术，但默认角色的分析框架能否自然迁移？能 → 用默认角色。

## 模式三：少于 4 角色、各代表一种成本
**来源**：`examples/three-roles-architecture-decision.md`
**触发信号**：议题有 2-4 个明确的专业维度，维度间存在真实成本权衡
**角色搭配**：2-3 个自定义角色，每个代表一种成本视角
**核心价值**：角色间的分歧本质是"不同成本的权衡"而非"好 vs 坏"，裁决更有操作性
**自检**：能否用一句话说清每个角色"在乎的成本是什么"？能 → 用这个模式。

## 模式四：Pre-flight 侦察 + 事实注入
**来源**：`examples/tool-selection-three-way-with-preflight.md`
**触发信号**：工具选型、平台迁移、配置评估——候选方案可能"根本不可用"
**角色搭配**：任意（关键不是角色而是侦察步骤）
**核心价值**：`which tool && tool --version` 一条命令可能颠覆整个分析方向；事实注入让角色产出"查过文档的人才会说"的内容
**自检**：候选方案中是否有"未确认可用性"的选项？有 → 先侦察。

## 模式五：受众代言人（内容设计类）
**来源**：`examples/content-design-audience-proxy.md`
**触发信号**：内容面向明确但不在场的受众，创作者与受众存在认知落差
**角色搭配**：叙事/信息设计角色 + 目标受众代言人（带具体画像）
**核心价值**：代言人直接回答"我会不会走神""我能不能理解"，比理论推演更有决策力
**自检**：讨论中是否频繁出现"读者可能会…""用户应该能…"这类猜测？有 → 拉一个代言人进来。
